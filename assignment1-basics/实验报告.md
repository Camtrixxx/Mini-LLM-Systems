## 2. BPE 分词器

### Problem (unicode1): Understanding Unicode

**(a)** `chr(0)` 返回 Unicode 码点 0 对应的字符,即空字符 NUL(`'\x00'`)。

**(b)** `__repr__()` 返回带引号的转义形式 `"'\\x00'"`,而 `print` 输出它的"实际内容"——一个不可见的控制字符,屏幕上什么都看不到。

**(c)** 它作为一个不可见字符嵌在文本中:`"this is a test" + chr(0) + "string"` 的 repr 显示 `'this is a test\x00string'`,但 `print` 出来是 `this is a teststring`,两段文字中间看不出任何东西(但字符串长度和字节内容都包含它)。

### Problem (unicode2): Unicode Encodings

**(a)** UTF-8 对 ASCII 为主的文本每字符只占 1 字节,而 UTF-16/32 分别至少 2/4 字节且大量字节是 `\x00` 填充——同样的文本序列更长、字节分布更稀疏浪费;而且 UTF-8 无字节序问题,是互联网的主导编码(>98%),训练数据天然就是 UTF-8。

**(b)** 该函数逐字节独立调用 `decode("utf-8")`,而 UTF-8 是变长编码——多字节字符的每个单独字节都不是合法的 UTF-8 序列。例如 `"你".encode("utf-8") == b'\xe4\xbd\xa0'`,对 `b'\xe4'` 单独 decode 直接抛 `UnicodeDecodeError`。只有纯 ASCII 输入才能侥幸正确。

**(c)** `b'\xc0\xaf'`:`0xC0` 开头的两字节序列是"过长编码"(overlong encoding),UTF-8 规范明确禁止,任何合规解码器都会拒绝;类似地 `b'\xff\xfe'` 中 `0xFF` 根本不是合法的 UTF-8 起始字节。

### Problem (train_bpe): 见 `cs336_basics/bpe.py`(15/15 测试通过)

实现要点:GPT-2 正则预分词 + 按 `<|endoftext|>` 对齐的分块多进程统计;merge 循环增量维护 pair 计数和 pair→word 倒排索引,每次合并只更新受影响的词。

### Problem (train_bpe_tinystories)

**(a)** 在 TinyStories train(2.1GB)上训练 vocab=10000 的 BPE(64 进程):**总耗时 56.2s**(预分词 14.0s + merge 42.2s),峰值内存 **0.27GB**。最长 token 是 **`" accomplishment"`(15 字节)**——一个完整的常用英文单词(带前导空格),对一个由 GPT-4 生成的儿童故事语料来说非常合理(其余最长 token 如 `" disappointment"`、`" responsibility"` 也都是完整单词)。

**(b)** 单进程下预分词(正则匹配 2.1GB 文本)占绝对大头;用 64 进程并行化预分词后,瓶颈转移到**串行的 merge 循环**(42.2s vs 14.0s)——每次 merge 中扫描受影响词并更新 pair 计数是纯 Python 的逐词操作,无法向量化。

### Problem (train_bpe_expts_owt)

**(a)** 在 OWT train(12GB)上训练 vocab=32000 的 BPE(96 进程):总耗时 **138 分钟**(预分词 102s,merge 8190s——661 万个唯一 pre-token 让串行 merge 循环成为绝对瓶颈),峰值内存 **10.3GB**(要求 ≤12h、≤100GB ✓)。最长 token 是 64 字节的 **`ÃÂÃÂ...`(«Ã»«Â» 重复 32 次)**——这是网页文本被双重 UTF-8 编码产生的经典乱码(mojibake),在网络爬取语料中高频出现,BPE 忠实地把它学成了一个 token;紧随其后的是 `----...`、`____...`、`====...` 等分隔线。有意义,但也说明 OWT 数据比较脏。

**(b)** TinyStories 词表几乎全是常用英文单词和儿童故事高频词(词汇简单、风格统一);OWT 词表则包含更多专有名词、技术词、URL 片段、代码符号和长复合词,反映了网络语料的多样性和噪声。相同 vocab 预算下,OWT 的 token 平均更"通用",TinyStories 的 token 更"领域特化"。

### Problem (tokenizer): 见 `cs336_basics/tokenizer.py`(24/24 测试通过,与 tiktoken GPT-2 完全对齐)

### Problem (tokenizer_experiments)

**(a)** 各取 10 个文档:TinyStories 10K tokenizer 在 TinyStories 上压缩率 **4.01 bytes/token**;OWT 32K tokenizer 在 OWT 上 **4.51 bytes/token**(词表大 3.2 倍、单 token 覆盖更长的常见片段)。

**(b)** 用 TinyStories tokenizer 编码 OWT 样本:压缩率降到 **3.41 bytes/token**(比域内的 4.01/4.51 都差)——词表是从儿童故事学的,OWT 里的复杂词、专有名词、技术符号都得拆成碎片甚至单字节,同样的文本要多花 ~32% 的 token。

**(c)** 单进程吞吐量约 **2.1 MB/s**(纯 Python 实现)。按此速度,825GB 的 Pile 单进程需约 **107 小时**;用 96 进程并行约 **1.1 小时**(实测编码 12GB OWT train 时 96 进程聚合吞吐 ~150 MB/s,与此一致)。工业级 tokenizer(如 tiktoken 的 Rust 实现)单核即可快 1-2 个数量级。

**(d)** 已将 4 个数据集编码为 uint16 npy:TinyStories train **541,229,347** tokens、valid **5,465,883** tokens;OWT train/valid 见 `out/data/`。uint16 合适是因为词表大小(10000/32000)都 < 2¹⁶ = 65536,每个 token 只需 2 字节,比 int64 省 4 倍磁盘/内存,配合 `np.memmap` 大幅降低训练时的数据加载开销。

## 3. Transformer 架构(实现见 `cs336_basics/model.py`,13/13 测试通过)

### Problem (transformer_accounting)

**(a)** 参数量 = 2·V·d(token embedding + lm_head)+ N·(4d² + 3d·d_ff + 2d) + d。GPT-2 XL 规格(V=50257, N=48, d=1600, d_ff=4288):**1.640B 参数**,fp32 加载需 **6.56GB**。

**(b)** 前向的矩阵乘(序列长 L=1024,单条序列):
| 矩阵乘 | 每层 FLOPs | 48 层合计 |
|---|---|---|
| QKV 投影(3 个 (L,d)×(d,d)) | 6Ld² | 0.755 TF |
| 注意力分数 QKᵀ((h,L,dₕ)×(h,dₕ,L)) | 2L²d | 0.161 TF |
| 加权求和 AV | 2L²d | 0.161 TF |
| 输出投影 (L,d)×(d,d) | 2Ld² | 0.252 TF |
| FFN w1/w3((L,d)×(d,d_ff)×2) | 4Ld·d_ff | 1.348 TF |
| FFN w2((L,d_ff)×(d_ff,d)) | 2Ld·d_ff | 0.674 TF |
| lm_head (L,d)×(d,V) | — | 0.165 TF |

总计:**≈3.52 TFLOPs**(单序列前向)。

**(c)** **FFN 占大头(57.5%)**,其次是注意力的 QKV/输出投影(28.6%);注意力分数计算只占 9.2%,lm_head 4.7%。

**(d)** 各规格前向 FLOPs 占比:

| 组件 | small (124M) | medium (350M) | large (774M) | XL (1.6B) |
|---|---|---|---|---|
| attn 投影 | 19.9% | 24.8% | 27.3% | 28.6% |
| attn 分数(QKᵀ+AV) | 13.3% | 12.4% | 10.9% | 9.2% |
| FFN | 39.8% | 50.1% | 54.3% | 57.5% |
| lm_head | 27.1% | 12.7% | 7.4% | 4.7% |

模型越大,**FFN 和 attn 投影(∝d²)占比越高**,而 lm_head(∝V·d,V 固定)和注意力分数(∝L²d,L 固定)占比越低——d² 项随宽度二次增长,支配了大模型的计算。

## 4-5. 训练基础设施

### Problem (cross_entropy / gradient_clipping / adamw / learning_rate_schedule / data_loading / checkpointing)

实现见 `cs336_basics/nn_utils.py`、`cs336_basics/optimizer.py`,全部测试通过。

### Problem (learning_rate_tuning)

玩具 SGD(lr/√(t+1) 衰减,10 步,loss = mean(w²)):lr=10 时 loss 从 26.3 缓慢降到 3.5;lr=100 时急速衰减到 ~10⁻²³(对这个二次目标一步就近乎到最优);lr=1000 时**发散**(loss 涨到 10¹⁸)——过大的学习率使更新步越过最优点且不断放大。

### Problem (adamw_accounting)

**(a)** 设 P 为参数量(见 3(a)),B=batch_size,L=ctx,N=层数,h=头数,d_ff=8d/3,float32:
- 参数:4P 字节;梯度:4P;优化器状态(m+v):8P
- 激活(按题目列出的组件):每块 7BLd + 4BL·d_ff + 2BhL²(RMSNorm×2、QKV 输出、QKᵀ、softmax、AV、输出投影、FFN 各中间量),外加最终 RMSNorm BLd、logits BLV、CE 的 softmax BLV,共 ×4 字节
- **总计 = 16P + 4·[N(7BLd + 4BL·d_ff + 2BhL²) + BLd + 2BLV] 字节**

**(b)** GPT-2 XL 代入:**总内存 ≈ 16.06·B + 26.25 GB**(26.25GB 是 16P 的参数/梯度/优化器状态)。80GB 限制下最大 batch size = **3**。

**(c)** AdamW 每步对每个参数做常数次逐元素运算(m、v 更新约 7 FLOPs,偏差修正步长为标量,参数更新约 5 FLOPs,权重衰减 2 FLOPs),合计 **≈15P FLOPs**(数量级 O(P),≈2.5×10¹⁰ 对 XL)。

**(d)** 总训练 FLOPs = 400000 步 × 1024 seq × 3.52 TF(前向)× 3(反向=2×前向)≈ **4.32×10²¹ FLOPs**。在 495 TF/s × 50% MFU 下 ≈ 1.746×10⁷ 秒 ≈ **4850 小时 ≈ 202 天**——单卡训 GPT-2 XL 完全不现实,这就是需要多卡并行的原因。

### Problem (experiment_log)

### Problem (learning_rate)

基线配置(vocab 10K / ctx 256 / d_model 512 / d_ff 1344 / 4层16头 / batch 128 / 10000 步 = 327.68M tokens,AdamW β=(0.9,0.95)、wd=0.1、warmup 500、cosine 到第 10000 步衰减到 lr/10,bf16 + torch.compile,单张 A800 每轮约 13 分钟)。

**(a)** 搜索策略:以 1e-3 为中心的对数网格,8 个 LR 并行各占一张卡,一轮扫完;发现最优在右端后又向上扩展了一轮。最终 val loss:

| lr | 1e-4 | 3e-4 | 5e-4 | 1e-3 | 2e-3 | 3e-3 | **5e-3** | 1e-2 | 2e-2 | 3e-2 | 5e-2 | 1e-1 | 3e-1 |
|---|---|---|---|---|---|---|---|---|---|---|---|---|---|
| val loss | 1.794 | 1.520 | 1.441 | 1.377 | 1.343 | 1.334 | **1.333** | 1.351 | 1.505 | 1.606 | 1.765 | 2.037 | **发散** |

最优 **lr=5e-3,val loss = 1.333 ≤ 1.45** ✓(曲线见 `out/plots/lr_sweep_val.png`)。

**(b)** lr=3e-1 在约第 1060 步 loss 冲上 >20 后 NaN,是硬发散;1e-2 到 1e-1 之间虽未发散但最终 loss 单调变差(bf16 + 梯度裁剪让训练"软着陆"而不是炸掉)。最优 lr(5e-3)正好处在"损失开始变差的拐点"左侧、距硬发散阈值约 60 倍——印证"最优学习率在稳定性边缘"的经验法则:更大的 lr 在发散前收敛更快,直到噪声/不稳定开始反噬。

### Problem (batch_size_experiment)

固定 10000 步,batch size 从 1 扫到 768(每个 batch size 单独调 lr;注意固定步数下大 batch 处理的总 token 数更多):

| batch size | 1 | 32 | 64 | 128 | 256 | 512 | 768 |
|---|---|---|---|---|---|---|---|
| 调后 lr | 1e-4 | 1e-3 | 2e-3 | 5e-3 | 7e-3 | 1e-2 | 1.2e-2 |
| val loss | 2.834 | 1.539 | 1.425 | 1.333 | 1.278 | 1.231 | 1.213 |

（注:原计划扫到 1024,但 batch=1024 × ctx 256 的 22.7M 模型 + `torch.compile` 在单张 A800-80GB 上激活显存 OOM,故上界取单卡可行的最大值 768。这本身也是一个实践观察:单卡显存直接限制了可探索的 batch 上界——再往上要靠梯度累积或多卡数据并行,后者正是 leaderboard 用到的手段。）

结论:batch=1 即使调低 lr 也几乎学不动(梯度噪声太大,单步信息量太小);增大 batch 每步梯度更准、可用更大的 lr,固定步数下最终 loss 单调改善,但边际收益递减(64→128 提升明显,512→768 已几乎持平)。实际训练在"每步耗时×收敛步数"之间取平衡,过大的 batch 浪费算力。曲线:`out/plots/batch_size_val.png`。

流畅度:语法正确、叙事完整、有简单的情节结构,达到了作业给的参考水平;偶有逻辑小瑕疵(如"给了他们那颗宝石"的因果略突兀)。影响质量的因素:①解码参数——temperature/top-p 截断低概率尾部,明显减少胡言乱语;②训练充分度与数据分布——模型只见过儿童故事,风格单一但域内极其流畅;此外模型容量(22.7M)限制了长程一致性。

### Problem (layer_norm_ablation)

去掉所有 RMSNorm 后,在最优 lr=5e-3 下**第 300 步即 NaN 发散**;降低 lr 能勉强稳定但效果惨烈:lr=3e-4 → val loss 5.73(基本没学会),lr=1e-4 → 1.993(能学但远差于基线 1.333)。RMSNorm 对 Transformer 的作用不只是"稳定训练",它通过控制每层激活的尺度,允许使用大一个数量级以上的学习率——没有它,要么发散要么被迫用小 lr 导致收敛又慢又差。曲线:`out/plots/ablation_noln.png`。

### Problem (pre_norm_ablation)

post-norm(z=RMSNorm(x+Attn(x)))在相同 lr=5e-3 下能训练,最终 val loss **1.397 vs pre-norm 1.333**,且训练前期 loss 下降明显更慢(残差主干被 norm 打断,梯度传播不如 pre-norm 干净)。这解释了为什么 pre-norm 成为现代 LLM 的共识选择。曲线:`out/plots/ablations_val.png`。

### Problem (no_pos_emb)

NoPE(完全去掉位置信息)最终 val loss **1.395 vs RoPE 1.333**。差距存在但出乎意料地小——因果 mask 本身让不同位置的 token 看到的上下文数量不同,decoder-only 模型确实能从中隐式推断位置;但显式的 RoPE 仍然收敛更快、最终质量更好。曲线:`out/plots/ablations_val.png`。

### Problem (swiglu_ablation)

参数量匹配(SwiGLU d_ff=1344 三个矩阵 vs SiLU d_ff=2048 两个矩阵)时:SiLU FFN val loss **1.336 vs SwiGLU 1.333**——在这个规模上两者几乎打平,门控的收益在小模型/小数据上不明显。这也印证了讲义的观点:很多架构选择的收益要到更大规模才能分辨,小规模消融要谨慎外推。曲线:`out/plots/ablation_swiglu.png`。

### Problem (main_experiment)

用与 TinyStories 完全相同的架构和训练步数(ctx 256 / d_model 512 / d_ff 1344 / 4层16头 / batch 128 / 10000 步,仅词表换成 OWT 的 32K)在 OpenWebText 上训练。先做了一轮 LR 试探(2e-3 / 5e-3 / 1e-2,各一张卡):

| lr | 2e-3 | **5e-3** | 1e-2 |
|---|---|---|---|
| 最终 val loss | 3.982 | **3.947** | 3.971 |

最优仍是 **lr=5e-3**(与 TinyStories 一致,且不在网格边界,是真正的谷底),最终 **val loss = 3.947**,单卡 21 分钟(曲线见 `out/plots/owt_main_val.png`,LR 对比见 `out/plots/owt_lr_sweep.png`)。

**损失差异的解读:** OWT 的 val loss(3.95)远高于 TinyStories(1.33)。但**两者不能直接比大小**——它们用的是不同的分词器(10K vs 32K 词表),每个 token 承载的信息量和先验熵不同,32K 词表下随机基线的交叉熵本就更高。抛开这一点,OWT 确实是**本质上更难**的建模任务:TinyStories 是用受限词汇、模板化句式合成的儿童故事,分布窄、熵低,22.7M 的小模型 10000 步就能把它学到接近流畅;OWT 是真实网络爬取文本,主题、语域、风格、噪声(乱码、代码、表格)极其多样,同样的模型容量和算力预算根本不足以覆盖这个分布,所以停在了高得多的 loss 上。换句话说,loss 的绝对值既反映了数据固有的不可约熵,也反映了"模型容量 × 训练量"相对任务难度的不足。

**生成文本**(temperature=0.8、top-p=0.9,完整 3 段见 `out/generation_owt.txt`):

> The vernacular is the second most popular side of the coin. It is usually considered the most popular feature of the coin, but it is rarely used in the coin. [...] The coin is used in the coin's creation. It is used in the coin's creation. It is used in the coin's creation. [...] as a symbol for the coin's creation.

**流畅度分析:** 与 TinyStories 输出形成鲜明对比。**局部**看语法基本正确、短语搭配自然("the second most popular"、"as a symbol for"),说明模型学会了英语的表层统计规律;但**全局**语义空洞、逻辑不自洽,并陷入严重的**短语级重复循环**("the coin's creation" 连续复读十余次)、主题无锚点漂移。

**为什么同样的模型和算力预算,OWT 输出质量明显更差?** 核心不在解码,而在**任务难度与模型/数据预算的错配**:①TinyStories 分布窄,10000 步足以让小模型逼近其"真实分布",于是采样出的故事既通顺又连贯;OWT 分布极宽,同样预算下模型只学到了 n-gram 级的局部流畅,远未学到段落级的语义规划和长程一致性,采样时就退化成"局部合理、全局胡言"并靠复读来维持局部概率。②OWT 数据本身更脏更杂(见 tokenizer 部分那些乱码 token),进一步稀释了有效学习信号。这正印证了讲义的观点:在固定小预算下,数据分布的复杂度直接决定了模型能达到的流畅度上限——要在 OWT 上达到 TinyStories 那种流畅,需要大得多的模型和多得多的训练量。

### Problem (leaderboard)

**最终验证 loss:3.2594**(OWT valid),学习曲线(wall-clock x 轴,42.2 分钟 < 45 分钟规则上限)见 `out/plots/owt_leaderboard_wallclock.png`;完整生成文本见 `out/generation_owt_leaderboard.txt`。

**预算折算:** 规则是"B200 上 ≤45 分钟"。本机是 8×A800-80GB,单卡 bf16 算力约为 B200 的 1/7,故用 **8 卡数据并行(DDP)跑 44 分钟**来逼近"1 张 B200 × 45 分钟"的等效算力,同时交付曲线的 wall-clock 轴天然 < 45 分钟,直接满足交付要求。DDP 用 `torchrun` 实现(见 `scripts/train.py`:每 rank 用 `seed+rank` 采不同 batch、梯度 all-reduce 平均、仅 rank0 记录/存档、eval 全 rank 对称执行以避免 collective 死锁)。

**在基线上做的改动(相对 main_experiment 的 22.7M / 单卡 / 10000 步):**

1. **权重绑定(weight tying)** — 输入 embedding 与输出 lm_head 共享权重(Vaswani et al. 2017 §3.4,PDF 明确推荐),并按提示调小初始化 std(0.02)。TinyStories 上对照验证:tie 后 val **1.312 vs 基线 1.333**,不但不掉点反而更好,且省下 `vocab×d_model` ≈ 24.6M 参数。
2. **放大模型到 GPT-2 small 级** — d_model 512→768、层数 4→12、头数 16→12、d_ff 1344→2048,共 **109.5M 参数**。45 分钟 × 8 卡的算力足以喂饱这个规模。
3. **多卡大 batch** — per-GPU batch 256 × 8 卡 = **有效 batch 2048**(单卡实测 batch 256 峰值 65GB,320 OOM,故取 256)。
4. **重新调 LR** — 换架构和有效 batch 后旧的 5e-3 不再适用。在正式 run 的完整配置(8 卡 / eff-batch 2048)上做 1200 步短扫,得到清晰的 U 型:

   | lr | 2e-3 | **3e-3** | 4.5e-3 | 6e-3 | 1.2e-2 | 2e-2 |
   |---|---|---|---|---|---|---|
   | 1200 步 val | 3.610 | **3.593** | 3.633 | 3.756 | 6.571 | 7.087 |

   最优 **lr=3e-3**(真正的谷底,两侧皆更差;≥1.2e-2 在大 batch 下直接失稳)。曲线见 `out/plots/owt_leaderboard_lr_sweep.png`。

正式 run:lr=3e-3、warmup 200、cosine 衰减到 5200 步(lr/10)、bf16 + `torch.compile`,44 分钟内跑完 **5500 步 ≈ 2.9B token(约 OWT 训练集一遍)**。

**效果:** val loss 从 main_experiment 的 **3.947 → 3.259**(同为 OWT、同 32K 词表,可直接比较),降幅 0.69。生成文本的长程一致性也肉眼可见地改善:

> The llamas are doing a pretty good job of that. They're finding the food," he said. [...] "I want to be able to tell them about my food," said the mother. [...] The llamas aren't just for show, either. They are a part of the family. [...]

相比 main_experiment 那段"the coin's creation"无限复读,leaderboard 模型能在整段里维持一个连贯主题(llamas / 母亲 / 家庭 / 食物)、实体一致、并保持对话结构——虽然仍偶有重复且逻辑不够严密,部分种子上还会退化成 token 循环,但段落级的语义规划能力明显强于小模型。这印证了在 OWT 这种宽分布数据上,**模型容量 × 有效训练 token** 是流畅度的主要瓶颈:同样 45 分钟预算下,把算力投到"更大模型 + 更大 batch + 权重绑定 + 调准 LR",能把 val loss 显著压低。

**注意事项(呼应讲义):** 这些在小规模 leaderboard 上有效的改动(尤其"放大模型"),未必按同样比例迁移到更大规模的预训练——权重绑定、最优 LR、batch/模型比例都会随规模漂移,小规模消融的结论要谨慎外推,这将在 scaling laws 单元进一步展开。
