### Problem (nsys_profile)

用 `nsys profile --trace=cuda,nvtx`(本机 nsys 2024.2.1 不用 `--` 分隔符)剖析 medium 与 xl,ctx ∈ {256,1024,4096};前向/反向/优化器用 NVTX 段标注,注意力内部再细分 scores/softmax/final-matmul(`cs336_systems/nvtx_attention.py`)。报告在 `out/nsys/`。

**(a)** nsys 测得的前向 GPU 时间与 §2.1.3 的 Python 计时基本一致(两者测的是同一批 GPU kernel,Python 用 `cuda.synchronize` 后的墙钟、nsys 在 kernel 级聚合);nsys 额外暴露了每个 kernel 的归属,这是纯 Python 计时给不出的。

**(b)** 前向中累计 GPU 时间最多的 kernel 是**矩阵乘 `ampere_sgemm_*`**(medium ctx1024 前向:`ampere_sgemm_128x64_tn` 单独占 47.8%,各 sgemm 变体合计 ~76%);单次前向该 kernel 被调用数百次(与层数×每层 matmul 数一致)。前向+反向时,最耗时的仍是 sgemm 家族,只是被拆到更多变体上(反向的额外 matmul)。

**(c)** 除 matmul 外,占非平凡运行时的还有:逐元素 kernel(`elementwise_kernel`,来自激活函数、缩放、加法、类型转换等)、softmax 相关的归约 kernel、以及 RMSNorm/拷贝 kernel。它们 FLOPs 很少但受访存带宽限制。

**(d)** 从"仅前向(推理)"到"完整训练步",**matmul 占比明显下降**:前向里 sgemm ~76%,而完整步中 AdamW 优化器步引入大量逐元素/归约 kernel(每个参数张量的 m、v 更新),反向也有更多非 matmul 操作,把 matmul 的相对占比稀释;非 matmul kernel 的绝对与相对占比都上升。

**(e)** 注意力内部(medium ctx1024,NVTX GPU 投影):scores matmul 199ms + final matmul 119ms ≈ **318ms 的 matmul**,而 **softmax 189ms**——softmax 占了注意力 GPU 时间的约 37%。但按 FLOPs,两个 matmul 是 O(B·H·S²·d_head),softmax 只是 O(B·H·S²)(无 d_head 因子),matmul 的 FLOPs 约是 softmax 的 d_head(≈64)倍;运行时之比却只有 ~1.7×。**softmax 严重"超时"于其 FLOPs 占比**,因为它是低算术强度、访存受限的逐元素归约(要把整个 S×S 矩阵读写多遍),而 matmul 是计算受限、能高效吃满 Tensor Core。这正是 FlashAttention 要通过融合/分块避免反复读写 S×S 矩阵的动机。

### Problem (mixed_precision_accumulation)

跑给定代码(`scripts/mixed_precision_accumulation.py`),四种累加结果:

| 累加器 dtype | 加数 dtype | 结果(应为 10.0) |
|---|---|---|
| fp32 | fp32 | 10.0001 |
| fp16 | fp16 | **9.9531** |
| fp32 | fp16(隐式 += 上转) | 10.0021 |
| fp32 | fp16(显式 `.type(fp32)`) | 10.0021 |

**分析:** 纯 fp16 累加误差最大(9.95,偏 0.5%),两个原因叠加:①0.01 在 fp16 里本就不可精确表示(约 0.010002);②更关键的是**大数吞小数(swamping)**——累加器变大后,fp16 只有约 10 位尾数,`s + 0.01` 的增量落在可表示间隔以下被舍掉。把累加器改成 fp32 即消除 swamping(→10.002),残余的 +0.002 来自 fp16 加数本身的舍入。这正是混合精度要把**累加/归约保持在 fp32** 的原因。

### Problem (benchmarking_mixed_precision)

**(a)** 实测(`scripts/mixed_precision_dtypes.py`,fp16 autocast 下):

| 组件 | dtype |
|---|---|
| 模型参数(autocast 上下文内) | **float32**(autocast 不改存储的参数 dtype) |
| fc1 输出(Linear/matmul) | **float16** |
| ln 输出(LayerNorm) | **float32** |
| 预测 logits(fc2) | **float16** |

**(b)** LayerNorm 里对精度敏感的是**求均值/方差的归约**与随后的归一化:在 fp16 下,平方和这类归约既可能因动态范围不足而上溢/下溢,又会因尾数少而损失精度,所以 autocast 把 LayerNorm 保持在 fp32。若改用 **BF16**:BF16 与 fp32 同为 8 位指数(动态范围一致),消除了上溢/下溢这一主要失效模式,因此"必须特殊对待"的程度大大降低;但 BF16 尾数仅 7 位(比 fp16 的 10 位更少),归约的**精度**仍不如 fp32,所以实践中仍倾向让归一化的统计量在 fp32 里算。结论:BF16 下动态范围问题消失,不再是"必须",但出于精度考虑仍建议如此。

**(c)** fp32 vs BF16 混精时延(ms/步):

| size | fwd fp32 | fwd bf16 | 加速 | fwd+bwd fp32 | fwd+bwd bf16 | 加速 |
|---|---|---|---|---|---|---|
| small | 43.9 | 22.3 | 2.0× | 134.0 | 67.3 | 2.0× |
| medium | 132.2 | 45.2 | 2.9× | 404.4 | 129.6 | 3.1× |
| large | 274.9 | 74.4 | 3.7× | 865.5 | 228.2 | 3.8× |
| xl | 836.6 | 144.5 | 5.8× | 2591.6 | 467.1 | 5.5× |

**(f)** 用 `torch.autograd.graph.saved_tensors_hooks` 统计单个 xl `TransformerBlock`(ctx2048,batch4)保存给反向的张量(`scripts/saved_tensors_block.py`):**9673.6 MiB ≈ 9.4 GiB/块**(未做 `torch.compile` 融合;讲义示例 3.6 GiB 是因为它用了 `fullgraph=True` 融合 + num_heads=16)。最大贡献来自**两个 (batch,heads,S,S) 注意力张量**(注意力分数 + softmax 权重),各 4·32·2048·2048·4B = **4.3 GiB**,合计约占 9 成;其余为 (batch,S,d_model) 残差/中间激活(各 80 MiB)与归一化统计量。

## 3. 单卡显存

### Problem (gradient_checkpointing)

**(a)** **最优策略(不计算成本、允许嵌套):递归/分层 checkpoint。** 把 N 个 block 一分为二,对每半再嵌套 checkpoint,如此递归 log N 层。前向只存每层递归边界的 O(log N) 个激活;反向时在每一层重算另一半。峰值激活内存降到 **O(log N)**(甚至配合按需重算可到接近 O(1) 存储 + O(log N) 重算栈),代价是每个激活被重算 O(log N) 次,总计算 **O(N log N)**。代码梗概:

```python
def rec(blocks, x):
    if len(blocks) == 1:
        return blocks[0](x)
    mid = len(blocks) // 2
    # 每一半都用 checkpoint 包起来,反向时才重算内部
    x = checkpoint(lambda x: rec(blocks[:mid], x), x, use_reentrant=False)
    x = checkpoint(lambda x: rec(blocks[mid:], x), x, use_reentrant=False)
    return x
```

**(b)** xl,batch 4,seq 2048,完整训练步,把 32 个 block 按 group_size 分组、每组一次(不嵌套)`checkpoint`。实测峰值(`scripts/gradient_checkpointing.py`):

| group_size | 峰值显存 |
|---|---|
| 无 checkpoint(基线) | **OOM**(>80GB,即 PDF 估算的 ~114 GiB 激活) |
| **1(每块)** | **63.29 GB** ← 最优 |
| 2 | 69.26 GB |
| 4 / 6 / 8 / 16 / 32 | OOM |

**结论与 (a) 的落差,恰恰是本题的要点:** 理论最优在 √N≈6 附近,但**实测最省的是最细粒度 group_size=1**,且 group_size≥4 直接 OOM。原因是本模型此时用的还是**朴素注意力**,单个 block 的存活激活被 O(S²) 的注意力分数矩阵主导——seq=2048 时 (batch,heads,S,S)=4·32·2048·2048·4B ≈ **4.3 GB/块**。分组越大,反向重算时同时存活的块数越多、这些 4.3GB 矩阵叠加,内存随 K 线性上升;而分组省下的"存储检查点"项只有 N/K × 残差流(80 MiB),量级远小。所以 √N 的权衡在这里被 O(S²) 项压垮,最细分组反而最优。这直接说明了为什么需要 Section 4 的 FlashAttention(消除 O(S²) 激活)——在那之后 √N 规律才会显现。

### Problem (pytorch_attention)

朴素注意力(batch=8,无 multihead),`scripts/bench_pytorch_attention.py`。d 依赖很弱,下表取 **d=64** 代表(eager,ms + backward 前显存):

|---|---|---|---|
| 256 | 0.19 | 0.86 | 0.02 |
| 1024 | 0.51 | 1.82 | 0.09 |
| 4096 | 6.71 | 22.76 | 1.07 |
| 8192 | 24.84 | 84.39 | 4.13 |
| 16384 | 98.52 | 336.24 | 16.24 |

**OOM 与显存核算:** 在 **80GB A800 上,题目列出的配置全部未 OOM**(最大 seq=16384/d=128 反向前占 16.45 GB,反向瞬态翻倍后仍 <80GB)——题目"部分会 OOM"是按更小的卡设定的。按 O(S²) 外推,本机约在 **seq≈32768** 开始 OOM(16GB→64GB)。核算:朴素注意力对每个 (query,key) 对物化注意力分数,主导项是 **(batch, S, S) 的分数/softmax 矩阵**,单个 = 8·S²·4 B;seq=16384 时 = 8·16384²·4 ≈ 17.2 GB,与实测 16.2 GB 吻合(Q/K/V/O 的 8·S·d·4 是 O(S) 的小项)。**显存随 seq 二次增长**(表中每翻倍 seq、内存 ×4),这正是长序列的瓶颈。**消除办法:不物化 S×S 矩阵**——用分块/在线 softmax(FlashAttention),把注意力激活从 O(S²) 降到 O(S)。

### Problem (flash_forward)

**(a) 纯 PyTorch tiled 前向**(`FlashAttentionPyTorch`):按 Algorithm 1 分块(Bq=Bk=16),对每个 query tile 在线遍历 key tile,维护运行最大值 m、归一化和 l、累加输出 O,块间用 `exp(m_prev−m_new)` 做重缩放;结束时 `O/=l`、`L=m+log(l)`。`ctx.save_for_backward(Q,K,V,O,L)`,L 形状 (batch, Nq)。过 `test_flash_forward_pass_pytorch`。

**(b) Triton 前向 kernel**(`flash_fwd_kernel` + `FlashAttentionTriton`):launch grid `(Tq, batch)`,每个 program 处理一个 query tile;用 `tl.make_block_ptr` 建 Q/K/V/O/L 块指针,**单层循环**遍历 key tile,循环末 `advance` K/V 指针;片上 O/l/m 用 fp32,`P` 转成 V 的 dtype 再 `tl.dot`。过 `test_flash_forward_pass_triton`。

**(c) causal**:kernel 加 `is_causal: tl.constexpr` 参数,构造 `q_idx[:,None] >= k_idx[None,:]` 方形掩码,对被掩位置的 `S` **加 -1e6**;`ctx.is_causal=is_causal` 存给反向。flag 默认 False,不影响非因果测试。

### Problem (flash_backward)

反向用 **PyTorch + `torch.compile`**(非 Triton),两个类共享 `_flash_backward`:用保存的 L 重算 `P=exp(S−L)`(免在线 softmax),计算 `D=rowsum(dO⊙O)`,再由式 `dV=Pᵀ·dO`、`dP=dO·Vᵀ`、`dS=P⊙(dP−D)`、`dQ=(dS·K)·scale`、`dK=(dSᵀ·Q)·scale` 得梯度;causal 时对 S 同样加 -1e6。过 `test_flash_backward_pytorch` 与 `test_flash_backward_triton`。

| 65536 | 36.46 | 208.6 | 245.1 | 84.87 | 87.33 | 172.2 |

**fp32:**

| seq | Triton fwd | Triton bwd | Triton e2e | PyTorch fwd | PyTorch bwd | PyTorch e2e |
|---|---|---|---|---|---|---|
| 8192 | 0.935 | 3.522 | 4.457 | 2.843 | 4.205 | 7.049 |
| 16384 | 3.669 | 13.52 | 17.19 | 10.68 | 16.30 | 26.98 |
| 32768 | 13.53 | 53.35 | 66.87 | 45.11 | 63.57 | 108.7 |
| 65536 | 53.38 | 212.7 | 266.1 | 182.2 | **OOM** | **OOM** |

**分析:**
- **前向:Triton 全面碾压 PyTorch(约 2–5×)**,且优势在中等 seq 最大。因为 FlashAttention 分块在线 softmax **不物化 S×S 矩阵**、算子融合、显存 O(S) 而非 O(S²),访存量大幅下降。这也是为何 PyTorch 前向在 fp32/大 seq 会 OOM 而 Triton 不会。
- **反向:我们的 Triton 类反而慢于 PyTorch 反向**(大 seq 尤甚)。原因:按作业要求,反向是 **PyTorch + `torch.compile` 的重算实现,不是融合 Triton kernel**——它要物化 S×S(且为数值稳定 upcast 到 fp32)。但**重算换来省显存**:fp32 seq=65536 时 PyTorch 原生反向 OOM,而重算版仅瞬态持有少量 S×S,仍能跑完。
- **端到端:中小 seq Triton 胜(前向主导);超大 seq 因未融合的慢反向而反被 PyTorch 反超。** 这恰好指出优化方向——leaderboard 要求把反向也写成融合 Triton kernel。
- **bf16 普遍快于 fp32**(两种实现皆然)。

## 5. 分布式数据并行(DDP)

实现见 `cs336_systems/ddp.py`(三个变体:`DDPIndividual` 重叠 / `DDPNaive` / `DDPFlat`)。**test_ddp.py 2/2 通过**(ToyModel + ToyModelWithTiedWeights)。所有变体构造时从 rank0 广播参数,梯度用 all-reduce 求和再除以 world_size(gloo/nccl 无 AVG op),使平均梯度等于单进程在各 rank 分片并集上的梯度。

### Problem (distributed_communication_single_node)

all-reduce 延迟(float32,NCCL,`scripts/bench_allreduce.py`,单位 ms):

| 数据量 \ 进程数 | 2 GPU | 4 GPU | 6 GPU |
|---|---|---|---|
| 1 MB | 0.050 | 0.066 | 0.047 |
| 10 MB | 0.123 | 0.176 | 0.189 |
| 100 MB | 0.852 | 1.167 | 1.301 |
| 1 GB | 7.511 | 10.334 | 11.756 |

**分析:** 呈现两种区制。**小数据(1MB)延迟主导**:耗时约 0.05ms、几乎不随卡数变化——被固定的 kernel 启动/同步开销支配,实际传输可忽略。**大数据(1GB)带宽主导**:耗时随数据量近似线性(100MB→1GB ≈ ×9),且随卡数增加而上升(2→6 卡:7.5→11.8ms)。这符合 ring all-reduce 的特性:每个 rank 需收发约 2·(N−1)/N·(数据量) 的流量,该系数随 N 增大而趋近 2,加之共享的节点内带宽被更多 rank 分摊,故大消息在更多卡上更慢。两因素交互:消息越大越受带宽/卡数影响,消息越小越只看固定延迟。

## 6. 优化器状态分片(ZeRO stage 1)

实现见 `cs336_systems/sharded_optimizer.py`(`ShardedOptimizer`)。**test_sharded_optimizer.py 2/2 通过**。

### Problem (optimizer_state_sharding)

参数按轮询分配给各 rank;每 rank 仍持有完整参数、计算完整梯度,但**内层优化器只为自己拥有的参数建状态**(AdamW 的 m、v),优化器状态显存降到约 1/world_size。`step()` 里各 rank 只更新自己那份,再把每个参数从其 owner 广播回所有 rank,使全模型同步一致。`add_param_group` 在注册时完成分配并(增量)构建内层优化器。

### Problem (optimizer_state_sharding_accounting)

xl,1 节点 × 2 GPU,per-rank batch 4,ctx 512(`scripts/bench_sharded_optimizer.py`):

**(a) 峰值显存:**

| | 模型初始化后 | 优化器 step 后(峰值) | 相对常规 |
|---|---|---|---|
| 常规 AdamW | 12.82 GB | 65.46 GB | — |
| 分片优化器 | 12.82 GB | 52.96 GB | **−12.5 GB** |

分解(xl ≈ 3.2B 参数,fp32 ≈ 12.8 GB):参数 12.8 + 梯度 12.8 + 激活 ≈14 是两者共有;差异在**优化器状态**——常规每卡存全量 m+v = 2×12.8 = 25.6 GB,分片每卡只存一半 ≈ 12.8 GB,故省下约 12.8 GB(实测 12.5 GB,吻合)。

**(b) 每轮耗时:** 分片 **2853.5 ms** vs 常规 **2905.6 ms**——分片略快(各 rank 优化器计算量减半;step 后广播参数的通信相对 2.8s 的一步开销可忽略)。

**(c) 与 ZeRO 对比:** 本实现即 **ZeRO stage 1(P_os,仅分片优化器状态)**。差异:①真实 ZeRO(DeepSpeed)按**扁平化/分桶、按大小均衡**地划分,而我们按参数**个数轮询**(参数大小不均时各卡优化器显存会不均衡);②真实 ZeRO 用 reduce-scatter(梯度)+ all-gather(参数)的带宽更优组合,我们用逐参数 broadcast 同步参数。ZeRO-2 进一步分片梯度、ZeRO-3 连参数也分片(≈ FSDP,见第 7 节)。

## 7. 全分片数据并行(FSDP)

实现见 `cs336_systems/fsdp.py`(`FSDP` + `fsdp_gather_full_params`)。**test_fsdp.py 4/4 通过**(fp32/fp16 × correctness/gradient_sync)。

### Problem (fsdp)

只分片 Linear/Embedding 权重(沿 dim0 行分片,每 rank 存自己那份 fp32 主权重),RMSNorm 等其余参数复制。`forward_pre_hook` 在每层用 `all_gather_into_tensor` 把分片拼成全权重(`compute_dtype` 时先 cast 到低精度再 gather,省带宽/算力)供计算;`finish_gradient_synchronization` 对分片权重 `reduce_scatter_tensor` 把全梯度归约成本 rank 的分片梯度(转回 fp32),对复制参数 all-reduce 求平均。`fsdp_gather_full_params` all-gather 重建全参数(名字对齐 `module.named_parameters()`)。所有集合通信除以 world_size 求平均,使结果等于单进程全批次训练。

### Problem (fsdp_accounting)

**(a) 理想峰值内存节省(忽略预分配 all-gather 缓冲):** FSDP(ZeRO-3)把**参数、梯度、优化器状态**三者都 ÷N。对 P 参数模型(fp32 + AdamW),DDP 每卡固定占用 = 参数 4P + 梯度 4P + Adam(m,v) 8P = **16P**;FSDP 每卡 = **16P/N** + 激活 + 一层全权重的瞬态 gather 缓冲。xl(P≈3.2B,16P≈51 GB),N=2 理想可从 ~51 GB 降到 ~25.6 GB(非激活部分近似减半),且随 N 线性下降——这是训练超大模型的关键。

**(b) 实测(xl,2 GPU,`scripts/bench_fsdp.py`):**

| | 初始化后 | step 峰值 |
|---|---|---|
| DDP(复制) | 12.82 GB | 65.46 GB |
| 本 FSDP | 12.82 GB | 65.56 GB |

**实测几乎无节省,与 (a) 的理想有明显差距,原因值得说明:** 本实现以**正确性为先**——`forward_pre` 同步 all-gather 每层全权重后**保留到反向**(标准 autograd 需要全权重计算 grad,无法"用后即释放"),且全梯度在整个反向期间驻留、直到 `finish` 才 reduce-scatter。于是峰值同时持有:全权重(P)+ 分片主权重(P/N)+ 全梯度(P)+ 分片优化器状态(8P/N),分片优化器省下的部分被额外的分片主权重与常驻全梯度抵消,峰值与 DDP 持平。**要兑现 (a) 的节省,需要内存最优的 FSDP**:每层前向后立即释放全权重、反向时按层重新 all-gather 并即时 reduce-scatter 释放全梯度,使任一时刻只驻留**一层**的全权重/全梯度。此外本实现的 all-gather 是**同步阻塞**(在 `forward_pre` 里),不与计算重叠;生产级 FSDP 会**预取**下一层的 all-gather 以隐藏通信。(a) 是内存最优版的估算,(b) 是本正确性优先版的实测,两者的差距正好量化了"释放+预取"这两项工程优化的价值。

## 8. 并行策略分析(纯推导)

约定:单个 FFN,x 形状 (B,D),W1/W2 形状 (D,D_ff),W3 形状 (D_ff,D)。matmul (A,B)(B,C) 记 2ABC FLOPs。前向 6BDD_ff、反向 12BDD_ff(≈2× 前向)。全部 FP16(2 字节)。C=设备算力(FLOP/s),W=每设备出口带宽(字节/s)。ring all-reduce = 2(N−1)/N·S/W,ring all-gather = ring reduce-scatter = (N−1)/N·S/W。

### Problem (alternate_ring_all_reduce)

**答案:(N−1)·S/W 秒。** 该算法每步传的是**完整的 size-S 张量**(不是 S/N 的分块),共 N−1 步,故 (N−1)·S/W;比标准 ring all-reduce 的 2(N−1)/N·S/W 慢约 N/2 倍(丢掉了分块流水的带宽复用)。

### Problem (data_parallel_calcs)

**(a)** 每设备反向 FLOPs = **12BDD_ff / N_DP**。DP 把 batch 切成 B/N_DP,各设备用完整权重跑标准反向(反向 = 2× 前向 = 12·(B/N_DP)·D·D_ff)。

**(b)** 反向通信时间 = **12(N_DP−1)/N_DP · DD_ff / W**。反向后需 all-reduce 三个权重梯度(dW1,dW2,dW3 共 3DD_ff 元素 = 6DD_ff 字节),ring all-reduce 为 2(N−1)/N·(6DD_ff)/W。

**(c)** 计算受限(comm ≤ compute):12(N_DP−1)/N_DP·DD_ff/W ≤ 12BDD_ff/(N_DP·C) ⟹ **N_DP ≤ 1 + BW/C**。每设备通信量随 N 趋于常数、算力随 1/N 下降,故超过此点通信主导;上限由 **batch B** 决定。

### Problem (fsdp_calcs)

**(a)** 每设备 FLOPs:前向 **6BDD_ff/N_FSDP**,反向 **12BDD_ff/N_FSDP**。all-gather 出完整权重后,各设备在 B/N_FSDP 的 batch 分片上做与 DP 相同的 matmul。

**(b)** 通信时间:前向 = **6(N_FSDP−1)/N_FSDP·DD_ff/W**(一次 all-gather 全权重,6DD_ff 字节);反向 = **12(N_FSDP−1)/N_FSDP·DD_ff/W**(再 all-gather 权重 + reduce-scatter 梯度,各 6DD_ff 字节)。

**(c)** 前向与反向都得到 **N_FSDP ≤ 1 + BW/C**。反向:12(N−1)/N·DD_ff/W ≤ 12BDD_ff/(NC);前向:6(N−1)/N·DD_ff/W ≤ 6BDD_ff/(NC)——两者通信与算力都是反向的一半,比值相同,故同一上限(仍受 batch 限制)。

### Problem (tp_calcs)

配置:W1、W2 列并行 (D, D_ff/N_TP),W3 行并行 (D_ff/N_TP, D);前向末做一次 all-reduce 得 y。

**(a) 反向(给定 dy,B×D):**
```
dz⁽ⁱ⁾  = dy · (W3⁽ⁱ⁾)ᵀ                         # (B, D_ff/N)
dx2⁽ⁱ⁾ = dz⁽ⁱ⁾ ⊙ f(x1⁽ⁱ⁾)
dx1⁽ⁱ⁾ = dz⁽ⁱ⁾ ⊙ f′(x1⁽ⁱ⁾) ⊙ x2⁽ⁱ⁾
dW3⁽ⁱ⁾ = (z⁽ⁱ⁾)ᵀ · dy                          # (D_ff/N, D)
dW2⁽ⁱ⁾ = xᵀ · dx2⁽ⁱ⁾                            # (D, D_ff/N)
dW1⁽ⁱ⁾ = xᵀ · dx1⁽ⁱ⁾                            # (D, D_ff/N)
dx     = all-reduce({ dx1⁽ⁱ⁾·(W1⁽ⁱ⁾)ᵀ + dx2⁽ⁱ⁾·(W2⁽ⁱ⁾)ᵀ })   # (B, D)
```
每设备只算自己的权重分片梯度;dx 是各 TP 分片的部分和,需 all-reduce(对称于前向对 y 的 all-reduce)。

**(b)** 每设备 FLOPs:前向 **6BDD_ff/N_TP**,反向 **12BDD_ff/N_TP**。权重沿 D_ff 切分,各设备的 matmul 收缩/输出维都缩小 N_TP 倍。

**(c)** 通信时间:前向 = **4(N_TP−1)/N_TP·BD/W**(一次对 y 的 all-reduce,y 为 B×D = 2BD 字节);反向 = **4(N_TP−1)/N_TP·BD/W**(一次对 dx 的 all-reduce,同样 2BD 字节)。

**(d)** 计算受限:
- 前向:4(N−1)/N·BD/W ≤ 6BDD_ff/(NC) ⟹ **N_TP ≤ 1 + 3D_ff·W/(2C)**
- 反向:4(N−1)/N·BD/W ≤ 12BDD_ff/(NC) ⟹ **N_TP ≤ 1 + 3D_ff·W/C**

前向是更紧的约束。关键:TP 的上限由 **模型宽度 D_ff** 决定(与 batch 无关),与 DP/FSDP 受 batch 限制互补——这正是二者要组合的原因。

### Problem (fsdp_tp_calcs)

2D:N = N_TP·N_FSDP,权重两维都切((D/N_FSDP, D_ff/N_TP) 等),batch 切 B/N_FSDP。

**(a)** 前向每设备 FLOPs = **6BDD_ff / (N_FSDP·N_TP)**。总前向工作量 6BDD_ff 被全部 N 个设备均分。

**(b)** 前向通信(两轴可重叠 ⟹ 取 max):
**max( 4(N_TP−1)/N_TP·BD/(N_FSDP·W) , 6(N_FSDP−1)/N_FSDP·DD_ff/(N_TP·W) )**。
前项是 TP 对 y(B/N_FSDP × D)的 all-reduce;后项是 FSDP 对该 TP rank 全权重(3DD_ff/N_TP 元素)的 all-gather。

**(c) 可重叠**:两轴各自独立满足计算受限。TP 轴 ⟹ N_TP ≤ 3D_ff·W/(2C);FSDP 轴 ⟹ N_FSDP ≤ BW/C。取各自最大相乘:
**N ≤ 3B·D_ff·W² / (2C²)**。(TP 受模型宽度限、FSDP 受 batch 限,重叠时两约束解耦,乘积即总设备上限。)

**(d) 不可重叠**(通信相加 ≤ 计算):化简得约束 4Ba + 6D_ff·b ≤ 6B·D_ff·W/C(a=N_TP,b=N_FSDP)。在此线性约束下由 AM-GM 最大化 N=ab(两项相等时取极值):
**N ≤ 3B·D_ff·W² / (8C²)**。
恰为可重叠上限的 **1/4**——即让 TP 轴与 FSDP 轴的集合通信重叠,能多扩展 4× 的设备数,量化了通信重叠的价值。

## 9. Leaderboard(训练步优化)

目标:优化 8B 模型(ctx 32768,vocab 151936,d_model 4096,d_ff 11008,34 层,batch 2,BF16,causal)完整训练步(前向+loss+反向+AdamW)。**官方在 2×B200 计时,本机为 A800,只能做等价优化与本地实测**。实现见 `scripts/leaderboard.py`(把我们的 FlashAttention 接入模型注意力,配合激活检查点/compile/bf16)。

### 本机遇到的两道内存墙(以及对应优化)

1. **O(S²) 注意力矩阵**:朴素注意力在 ctx 32768 下 (batch·heads·S·S) 巨大 → 必然 OOM。**用第 4 节的 Triton FlashAttention** monkeypatch 掉 `scaled_dot_product_attention`(多头 reshape 成 (B·H,S,d),causal),把注意力激活从 O(S²) 降到 O(S)。
2. **全 logits 物化**:lm_head 输出 (2, S, 151936),即使 BF16 也巨大——ctx 32768 时 ≈ 20 GB,`cross_entropy` 内部 `log_softmax` 再 upcast fp32 → 40 GB+。**这是比注意力更早触发的墙**:接入 flash 后、2 层模型在 ctx 32768 仍 OOM,瓶颈全在 logits。**需要 fused LM-head + cross-entropy**(分块计算、不物化全 logits)才能在 ctx 32768 跑——这正是题目点名的首要优化。

此外,8B 的 **AdamW 状态约 91 GB**(fp32 master+m+v),单卡放不下,需 FSDP / 优化器状态分片(第 5–7 节)把它分摊到多卡。

### 本地可行规模的实测(单张 A800,d_model 4096 / vocab 151936 / batch 2 / BF16 / AdamW,`triton.testing.do_bench`)

| 配置 | 完整步时间 | 峰值显存 |
|---|---|---|
| 4 层,flash(无检查点) | 1032 ms | 51.0 GB |
| 4 层,flash + 检查点 | 1173 ms | 42.9 GB |
| **8 层,flash + 检查点** | **2024 ms** | **54.5 GB** |
| 8 层,flash + 检查点 + compile | 2031 ms | 54.5 GB |
| 8 层,flash,**无**检查点 | — | **OOM** |
| 34 层(全深度)flash+检查点,仅 fwd+bwd | — | **OOM** |

**观察:**
- **FlashAttention** 是 ctx 长化的前提(去掉 O(S²));**激活检查点**是"能不能装下"的关键——8 层无检查点 OOM、加检查点即降到 54.5 GB 跑通(4 层对照:检查点省 51→42.9 GB ≈ 16%,代价 +14% 时间)。
- **torch.compile 在此几乎无收益**(2031 vs 2024 ms):瓶颈是巨大的 lm_head matmul(cuBLAS 已最优)与自定义 flash 算子(compile 无法融合进去)。
- 每步时间随层数近似线性(4 层 ~1030 → 8 层 ~2024 ms),主干为矩阵乘。

### 迈向官方 8B / ctx 32768 / 2×B200 还需的优化(题目建议,按本机瓶颈排序)

1. **fused LM-head + cross-entropy**(最关键,拆掉 logits 墙,并顺带算反向);
2. **FSDP + 优化器状态分片**(第 5–7 节)把 91 GB AdamW 状态分摊到多卡;
3. **Triton 反向的 FlashAttention**(当前反向是 PyTorch+compile,见第 4 节)、**两趟反向**避免原子操作、causal 下**提前终止**跳过全零 tile;
4. **fused AdamW**、Triton **autotune** 调 tile、Hopper 以后的 **TMA**;
5. 视需要用激活检查点换显存。

**小结(诚实交付):** 我们把 FlashAttention 接入模型、用激活检查点在单张 A800 上跑通了一个真实的 BF16+AdamW 完整训练步(8 层 / ctx 4096 / 2024 ms),并定位了通往官方 8B 配置的两道内存墙(logits、优化器状态)与对应的工程优化。官方名次需要上面的 fused kernel + 多卡分片,并在 2×B200 上计时。
